# Проект 16: Мультиагентная система CrewAI для создания контента

## Обзор архитектуры и проектирование агентов
В рамках данного проекта был разработан мультиагентный конвейер (pipeline) с использованием фреймворка CrewAI. Система моделирует работу реальной редакции и состоит из трех специализированных агентов:

1. **Старший аналитик-исследователь**: Отвечает за сбор сырых, но достоверных данных. Ему был выдан инструмент `SerperDevTool` для доступа к реальному веб-поиску. Это критически важное архитектурное решение, которое предотвращает "галлюцинации" LLM и позволяет оперировать актуальными фактами.
2. **Главный контент-райтер**: Принимает исследовательский бриф и трансформирует его в структурированную статью формата Markdown.
3. **Главный редактор**: Выполняет финальную вычитку, проверку логики и стиля, после чего автономно сохраняет готовый файл в директорию `output/`.

**Связь задач (Data Flow):** Задачи передаются строго последовательно (Process.sequential). Результат выполнения `research_task` (бриф) автоматически становится входным контекстом для `write_task`, а черновик райтера передается на вход `edit_task`. 

## Возникшие трудности и их решения
В процессе разработки я столкнулся с несколькими интересными инженерными вызовами:

1. **Конфликт пространства имен и архитектуры файлов**: Изначально скрипты нового проекта смешались со старыми файлами от RAG-системы, что привело к ошибке `ModuleNotFoundError` при поиске модулей `crew`. Проблема была решена строгой изоляцией Проекта 16 в отдельную директорию с чистой структурой импортов.
2. **Порядок инициализации переменных окружения**: Возникла ошибка `OPENAI_API_KEY is required`, несмотря на наличие файла `.env`. Анализ показал, что импорт агентов (которые немедленно стучатся к API OpenAI) происходил до вызова `load_dotenv()`. Решение: перенос инициализации переменных окружения на самую первую строку главного скрипта `main.py`.

## Идеи для будущих улучшений
Система показала высокую автономность, но имеет потенциал для масштабирования:
* **Внедрение иерархического процесса (Hierarchical Process)**: Вместо жесткой последовательности можно добавить агента-Менеджера, который будет сам распределять задачи между Райтером и Исследователем, оценивая их загруженность и качество промежуточных ответов.
* **Добавление агента SEO-оптимизатора**: Внедрение четвертого агента, который будет проверять финальный текст от Редактора на наличие ключевых слов и LSI-фраз перед публикацией.
* **Интеграция с базой данных**: Замена простого сохранения в `.md` файл на инструмент автоматической публикации черновиков напрямую в CMS (например, WordPress) или Notion через API.
